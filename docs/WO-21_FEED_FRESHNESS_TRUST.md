## 0. Executive Summary (답부터)

**2026-07-13(월) 코스피 -8.95% 폭락일에 앱 브리핑은 "+2.52% 상승"을 보여줬다.**
원인은 시세 소스가 아니라 **우리 파이프라인 3중 결함**: ① 크론 슬롯 오분기(지연 실행 → else→close 폴스루)로 장 시작 전 새벽에 "오늘의 국장"이 전 거래일 데이터로 생성, ② 시세 fetch의 스테일 캐시 허용, ③ 생성물에 거래일 검증 가드 부재.

처방은 3단계:
- **Phase 0 (즉시, 이번 PR)**: 정합성 — 슬롯 오분기 제거, 시세 fetch 강제 신선(no-store), 거래일 불일치 시 발행 차단(fail-closed), 지수 부호 파서 강건화, HOT ISSUE 중복 제거.
- **Phase 1 (금주)**: 메인 이벤트 대응 — 매크로 헤드라인을 브리핑 노트 입력에 연결(전쟁·금리 등 '왜'가 노트에 실림), 장중 급변 감지 카드(|코스피| ≥ 2% 시 장중 발행).
- **Phase 2 (차주+)**: 밀도 — 사전 정제(규칙/경량 분류) 후 LLM 해석 파이프라인(비용 ↓·품질 ↑), TradingAgents식 다관점 해석 구조 검토.

## 1. Situation — Complication — Question

**Situation.** 피드 콘텐츠(브리핑·버즈·핫이슈)는 크론이 빌드해 `FeedContentCache`에 쓰고, 요청 경로는 읽기만 한다(504 원칙). KR 시세는 네이버, 지수는 TwelveData→네이버→Yahoo 순.

**Complication.** 이란 전쟁 이슈로 코스피 -8.95% 폭락한 날:
1. 브리핑이 정반대(+2.52%/+5.47%) 수치를 노출 — 신뢰 상품인 "정직한 숫자 원칙"이 정면으로 깨짐.
2. 폭락의 '왜'(전쟁)가 피드 어디에도 없음 — 뉴스는 수집되고 있었는데(연합 market/economy RSS 존재) 표면화 실패.
3. HOT ISSUE 카드가 같은 신디케이트 기사 제목 3줄을 반복 — 정보 밀도 제로.

**Question.** 어떻게 (a) 다시는 시장과 반대인 숫자가 나가지 않게 하고, (b) 메인 이벤트가 반드시 표면화되며, (c) 카드 한 장의 정보 밀도를 높이는가?

```

    ├─ D1. 클러스터링이 신디케이트 중복 제목에 지배됨(제목 dedup 없음) → 전쟁 대신 "SNS 미담주" 3줄
```

### 2.3 왜 오늘 처음 티가 났나

이 구조에서는 **지연된 야간 크론이 있는 날마다 브리핑이 전 거래일 데이터가 될 수 있었다**
(7/10·7/11·7/12에도 21시대 close 오분기 실행 확인). 상승일에는 "어제 숫자"도 그럴듯해 보여서 발각되지 않았고, 방향이 뒤집힌 폭락일에 처음 노출됐다.

### 데이터 흐름 (Phase 0 후)

1. 입력: GH cron(정확 슬롯) → Vercel 크론 라우트.
2. 처리: 네이버 시세 **신선 fetch** → 거래일 가드(오늘 KST와 일치 검증) → 빌드 → LLM 노트(기존 가드 유지).
3. 출력: `FeedContentCache` upsert. 가드 실패 시 **발행하지 않고** 로그(스테일 카드 0장 원칙).
4. fallback: 거래일 불일치(휴장일 포함) → 브리핑 스킵 = 정직한 무카드. 다음 정상 크론이 재시도.

### 검증 게이트

- `npm run spec:analyze`, `npm run guard:discovery`(discovery-supply 접촉), 관련 vitest, typecheck/build.
- 수동: 머지·배포 후 `workflow_dispatch slot=close` 재실행 → `/api/fomo/feed-hub` 브리핑이 7/13 실측(-8.95/-4.55)과 일치 확인.
- 실패 시 머지 금지 조건: 기존 markets/feed 테스트 하나라도 깨짐, guard:discovery 회귀.

## 6. 리스크

- **정직한 무카드**: 가드 발동일엔 브리핑이 빠진다 — 틀린 숫자보다 낫다는 원칙(정직한 숫자)으로 수용. 발동 시 로그로 감지.
- **네이버 레이트리밋**: 크론 경로만 no-store(하루 수십 콜 증가) — 요청 경로 캐시는 유지.
- **GH cron 지연 자체**는 못 고침 — 슬롯 정확 매핑으로 "지연돼도 틀린 일은 안 함"을 보장. close 크론이 지연되면 브리핑이 늦게 나올 뿐 틀리게 나오지 않는다.
